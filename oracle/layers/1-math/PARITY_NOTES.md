## Purpose

`parity-fixtures.json` is the locked reference dataset that gates Bite 3 (extraction). For every fixture, the file contains:

- A snapshot of the inputs that went into `computeLambdaBase` (production code in `scripts/live/strikeoutEdge.js`)
- The full structured return of `computeLambdaBase` for those inputs (`expected_inner`)
- Production-logged outer-chain values (`expected_outer_chain_from_production`) — split_adj, opp_adj, park_factor, weather_mult, ump_factor, lambda_final
- Per-strike NB probabilities computed via `archetypeR(savant)` and production-logged `lambda_final`

When Layer 1 v1.0 ships in Bite 3, it must reproduce every numeric field exactly (numeric tolerance 1e-12; strings/bools exact).

## Cross-check policy

For each fixture, the harvester compares the recomputed `computeLambdaBase` output against production's logged values in `decision_pipeline.lambda_calc_json` and `model_input_json`.

**Fields checked (numeric, 1e-12 tolerance):**
- `lambdaBase` ↔ production `lambda_base`
- `pK_blended` ↔ production `p_k_blended`
- `veloAdj` ↔ production `velo_adj`
- `bbPenalty` ↔ production `bb_penalty`
- `ttoPenalty` ↔ production `tto_penalty`
- `expectedBF` ↔ production `expected_bf` (in `model_input_json`)
- `avgPitches` ↔ production `avg_pitches`

**Fields checked (exact, string/bool):**
- `ttoNote`
- `leashFlag`
- `bfSource` (in `model_input_json`)

## Limitations recorded for future versions

1. `fetchGameLog` hits the live MLB API and returns CURRENT season-to-date data. Fixtures capture a snapshot at fixture-build time, not a true historical replay. v1.1 enhancement: persist raw inputs at decision time.

2. `pitcher_statcast` and `pitcher_recent_starts` reflect current DB state. Pitchers who have pitched since their fixture's `bet_date` will produce drift in cross-check. The 6 mismatches in this run are all this category.

3. `manager_leash_factor` is null for all current 2026 pitchers (insufficient sample). When the field becomes populated mid-season, this fixture set should be regenerated to capture the `post_il_short_leash` archetype.

4. The harvester does NOT reproduce the outer chain — it reads production-logged values for split_adj/opp_adj/park_factor/weather_mult/ump_factor. Layer 1's outer-chain reconstruction is tested by feeding those same logged values + harvested lambda_base into Layer 1's outer-chain implementation and asserting it produces the same `lambda_final`.

### nb_r_source semantics

| Source | Meaning | Trigger | r value |
|---|---|---|---|
| `fitted` | Per-pitcher dispersion fit by `scripts/live/fitDispersion.js` and stored in `pitcher_statcast.nb_r`. | `savant.nb_r != null` | the fitted value |
| `archetype_kpct` | Bucketed by season K%. | `savant.k_pct != null` and no fitted `nb_r` | `≥ 0.28 → 20` (power), `≤ 0.19 → 50` (contact), else `30` (mixed) |
| `global_default` | No savant data. | `savant == null` or `savant.k_pct == null` and no fitted `nb_r` | `NB_R = 30` |

Branch precedence: **fitted > archetype_kpct > global_default**. This mirrors the body of `archetypeR()` in `lib/strikeout-model.js`.

### Layer 1 vs legacy live paths — known inconsistencies

The audit found three real inconsistencies in legacy live paths. **Documented, not fixed.** Fixing them would change live in-game probabilities and is gated on a calibration run (Bite 5b, deferred).

| # | File | Issue | Severity |
|---|---|---|---|
| A | `scripts/live/liveMonitor.js:376` | Uses `pK_blended >= 0.28 ? 20 : pK_blended <= 0.19 ? 50 : 30`. That's the same bucket cuts as `archetypeR` but applied to the **model-blended** K% (after L5/career/season blend + ML overlay), not raw `savant.k_pct`. Imports `archetypeR` but never calls it. | Wrong input variable; can land in different bucket than pre-game. |
| B | `scripts/live/liveMonitor.js`, `livePositions.js`, `inGameEdge.js` | Fitted `savant.nb_r` (from `fitDispersion.js`) is **never consulted** in any live path. Pre-game (strikeoutEdge.js → archetypeR) does honor it. | Fidelity loss for any pitcher with a fitted `nb_r`. |
| C | `scripts/live/livePositions.js:42`, `scripts/live/inGameEdge.js:282` | `livePositions.computeLiveProb` accepts an `nbR` parameter but the body calls `pAtLeast(lambdaRemaining, needed)` without passing it. Caller (`livePositions.js:244`) doesn't pass it either. `inGameEdge.js:282` calls `pAtLeast(live.lambda_remaining, needed)` without `r`. Both fall back to `NB_R = 30` for every pitcher. | Code bug; live always uses r=30 for power and contact pitchers. |

**Net effect of A+B+C combined**: the legacy live paths use a different (and in some cases just plain wrong) dispersion than the pre-game / Oracle Layer 1 path.

### Why we did not fix them in Bite 5

- Live trading is currently producing acceptable behavior. Changing dispersion for live without a backtest violates "don't break what works."
- The strategic direction is for **Oracle (Layer 2 Path / Layer 5 Judge / Layer 6 Gateway) to own live decisions**, not the legacy `liveMonitor` / `livePositions` / `inGameEdge` modules. Patching legacy live paths now is wasted work if the Oracle migration is going to replace them.
- Calibration tooling exists (`scripts/live/calibrateNB.js`, `scripts/live/backtest.js`'s NB_R sweep). Bite 5b — if pursued — should run the sweep using `archetypeR(savant)` per pitcher and measure Brier / log-loss delta vs current behavior before any live-path edit.

### Locked behavior (matches user spec)

- **BF-only blend.** `pK_ours` preserved. `lambda_base_blended = ((1-w_dk)×E[BF]_ours + w_dk×BF_dk) × pK_ours`. Outer chain runs untouched.

- **Missing DK data returns structured skip, not throw.** `dk_line` / `over_price` / `pK_ours` / `expected_bf_ours` / `r` invalid → returns object with appropriate `skip_reason`. - **Counterfactual visibility.** `dk_lambda`, `bf_dk`, `bf_delta` populated whenever DK inputs allow them, regardless of skip/apply. Lets shadow logging observe what the blend would have done.

### bfSource tier mapping (locked)

| Prefix in `bfSource` | Tier | Source quality |
|---|---|---|
| `BF×N` | strong | recent_starts table, real BF observations, N≥2 |
| `logBF×N` | medium | game-log derived (recent_starts had <2 rows) |
| `ip×PA/IP` | weak | synthesized BF from IP × constant |
| `career_ip×PA/IP` | weak | career fallback only |
| anything else / null | unknown | conservative |

Modifier suffixes (`(NNpc)`, `→capped(...)`, `×leash(...)`) ignored — only prefix matters.

### Vig caveat (recorded, not fixed)

`dk_k_props.over_price` is the book's implied probability and **includes vig**. The fetcher (`lib/odds.js`) requests `oddsFormat='decimal'` and stores `1 / decimal_odds`. Under price is not stored, so we cannot vig-adjust by the standard renormalization. Direction of bias: `over_price` overstates fair P(over) by ~2-3 percentage points → λ_dk biased upward by a few tenths of a K.

Bite 6 v1 accepts this; the Bite 6.3 backtest will show whether the signal survives the bias. v1.1 may add `under_price` to the schema and vig-adjust.
